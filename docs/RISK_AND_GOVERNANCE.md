# 风险控制与治理方案

## 1. 核心风险

| 风险 | 示例 | 当前控制 |
|---|---|---|
| 事实编造 | 1ms 延迟、300 小时续航、FDA 认证 | Prompt 约束 + Safety Rules + 事实一致性审核 |
| 竞品和品牌风险 | 直接提 Keychron、Razer | 禁用词过滤 + 品牌安全审核 |
| 合规风险 | 100% 无延迟、消除疼痛、保证效果 | 合规词表 + 合规风险审核 |
| EDM 合规缺口 | 缺少退订入口 | EDM 规则检查 |
| 引用误导 | 引用案例与 Brief 不相关 | Top 3 展示 + 人工抽检 |
| API key 泄漏 | 让面试官填 key | Streamlit Secrets + 隐藏 API 设置 |

## 2. 人机协同边界

系统负责：

- 生成初稿。
- 检索知识库引用。
- 根据固定维度自动审核。
- 给出迭代建议。
- 输出人审意见。

人工负责：

- 确认产品事实和参数。
- 确认目标市场邮件合规要求。
- 确认品牌表达和最终发布。
- 补充真实认证、保修、价格、用户评价等来源。

## 3. 内容发布前检查清单

### 通用

- 是否出现竞品词？
- 是否出现未提供的参数、认证、保修、折扣？
- 是否出现用户评价、百分比、销量等未提供数据？
- 是否出现 best、ultimate、guaranteed、100% 等绝对化表达？
- CTA 是否符合营销目标？
- Top 3 引用是否与 Brief 相关？

### EDM

- 是否有 unsubscribe / manage preferences？
- 发件人和落地页是否真实？
- 是否符合目标市场邮件规则？
- 是否避免过度促销和虚假优惠？

### Blog

- 是否混入 EDM 退订页脚？
- 标题和小标题是否适合 SEO？
- 是否需要配图或产品链接？
- 是否有清晰但不过度促销的 CTA？

## 4. 知识库治理

新增知识入库前需要检查：

- 来源是否可信。
- 是否标记内容类型：Blog / EDM / product_fact / compliance / brand_voice。
- 是否包含可检索标题和正文。
- 是否过期，例如价格、促销、保修政策。
- 是否含有未经授权的竞品表达。

## 5. 低分补库与失败回流

低分维度补库机制：

1. 每次测评后读取 `eval_details_*.jsonl`。
2. 将终审维度分数 `<= 7` 的问题写入 `knowledge_gap_backlog_*.csv`。
3. PM 根据维度判断补充类型：产品事实、案例、品牌规范、合规规则或渠道模板。
4. 更新 Qdrant 知识库，并确保 payload 中有 `type`、`title`、`content` 和可检索标签。
5. 重新运行 rules 全量测评和 API 小样本测评，对比低分维度是否减少。

线上失败案例回流机制：

1. 记录真实失败 Brief、错误输出片段、失败维度和期望行为。
2. 使用 `online_failure_case_template.json` 生成候选用例。
3. 人工确认该问题可复现、可验收、不是临时 API 或网络故障。
4. 将确认后的用例追加到 `marketing_testset.jsonl`。
5. 修复 Prompt、规则或知识库后，将该失败案例作为固定回归用例。

## 6. 升级方向

- 引入人工审核状态：待审、通过、驳回、需补资料。
- 对每次低分维度做标签沉淀，反哺知识库。
- 对产品事实建立版本号，避免旧参数被引用。
- 建立敏感表达词表后台维护能力。
- 将最终人工通过内容回流为高质量案例。
